# caselens(判例透鏡)

**台灣法院判決書個人版檢索 — 兩階段 retrieval(keyword + LLM re-rank)→ 賠償區間預估 + 實務建議。**

民眾遇到車禍 / 鄰居糾紛 / 勞資爭議 / 房東房客 → 用自然語言描述情境 → caselens 找出 Top 3 最相似判決 + 純函式聚合賠償統計 + LLM 預估你這案的賠償區間 + 蒐證清單 + 下一步建議。

把「**律師諮詢一次 NT$3-5K 才能知道法院通常怎麼判**」變成「**30 秒看到 3 個最相似的判例 + 預估你的賠償區間 NT$23-45 萬**」。

---

## 痛點

民眾每年遇上**車禍 / 鄰居噪音 / 房東惡意扣押金 / 商家拒退貨**等糾紛,**最迫切想知道**:「我這種情況法院通常會判多少?」「對方保險公司給 NT$8 萬合理嗎?」「值不值得花 NT$5K 諮詢律師?」

> **「車禍對方保險公司只給 NT$8 萬,我問律師說至少 NT$30 萬以上,但諮詢費要 NT$3K 我不確定要不要花。」** — Dcard 法律板真實案例

> **「想自己上司法院判決書搜尋,2 個小時看了 30 個案件還是不知道我的情況對標哪個。」** — PTT Law 板

> **「LegalBank 月費 NT$3K 給律師用,我們普通人查一次需要花太多。」** — FB 車禍受害者社團

具體痛點:

- 司法院裁判書檢索系統**介面是 2005 年水準**;搜尋結果一堆無關案件
- LegalBank / 法源 / Wisers 偏律師工具,**民眾沒辦法低價單次使用**
- 律師諮詢一次 NT$3-5K,**單純問「會判多少」就要花 5K 太貴**
- 法扶基金會排隊 3-7 天才能諮詢
- **ChatGPT 直接問會幻覺嚴重**(編造判決字號 / 賠償金額)
- 民眾不知道「揮鞭症候群」/「精神慰撫金實務區間」/「過失比例認定」等法律術語應該對應什麼判例
- 對方保險公司的「**低額快速和解**」是民眾權益最大殺手,但民眾沒法律憑據去談判

## 為什麼現在沒有對的工具

| 既有工具 | 為什麼不行 |
|----------|-----------|
| **司法院裁判書檢索** | 公開但介面糟、搜尋結果無排序、不抽要旨、不算賠償統計 |
| **LegalBank / 法源** | 月費 NT$3K+,給律師專業用;民眾一次性查不到性價比 |
| **律師諮詢一次 NT$3-5K** | 用「問會判多少」太貴;且律師多半引導你委任他打官司 |
| **法扶基金會免費諮詢** | 排隊 3-7 天;每月名額有限 |
| **ChatGPT / Claude 直接問** | 幻覺嚴重(編造判決字號 / 賠償金額);不知即時案例 |
| **網路法律 KOL 文章** | 偏「教科書式」說明,**沒有對標你個案的相似判例** |
| **Wisers 法學資料庫** | 學術用,介面更糟,且不是 SaaS |

**Gap 結構性**:**Vector retrieval + LLM re-rank** 對中文判決書是天然契合 — 但司法院 / LegalBank 都是「搜尋系統」而非「個案匹配系統」。Google「台灣 法院判決 個人版 檢索」零中文 SaaS 結果。

## caselens 在做什麼

```
使用者自然語言情境描述(.txt)
        │
        ▼
  ┌──────────────────────────────────────────────────────────┐
  │ Stage 1: 純函式 keyword Jaccard 相似度檢索                  │
  │ (retriever.py)                                            │
  │   - extract_legal_keywords(文字 → set)                    │
  │   - 同義詞展開:「沒戴安全帽」→ 未戴安全帽                  │
  │   - 對 corpus 中每個 case 算 Jaccard                       │
  │   - 取 Top 8 候選                                          │
  │ 真實產品:用 sentence-transformers (BAAI/bge-zh) 算 dense   │
  │           embedding;此 prototype 用 sparse Jaccard 簡化     │
  └────────────────┬─────────────────────────────────────────┘
                   │
                   ▼
  ┌──────────────────────────────────────────────────────────┐
  │ Stage 2: Claude LLM Re-rank by 法律相關性                  │
  │ (caselens.py:llm_rerank_and_explain)                      │
  │   注意 keyword 抓不到的法律語意:                            │
  │     - 角色方向(我被追撞 vs 我追撞別人)                     │
  │     - 違規嚴重度(酒駕 / 闖紅燈 vs 一般未注意)               │
  │     - 過失比例(0% vs 50%+ 對賠償差很多)                    │
  │     - 傷勢輕重(輕傷 / 骨折 / 粉碎性骨折 / 重傷)             │
  │   → Top 3 相關性最高案件 + 為什麼相似                       │
  └────────────────┬─────────────────────────────────────────┘
                   │
                   ▼
  ┌──────────────────────────────────────────────────────────┐
  │ 純函式聚合 Top 3 賠償統計:                                 │
  │ (retriever.py:compute_compensation_stats)                 │
  │   - min / median / max / avg 賠償金額                      │
  │   - 原告過失分布(0% / 部分 / 主因)                          │
  │ **LLM 永不算 NT$** — 全部從 case data 引用                 │
  └────────────────┬─────────────────────────────────────────┘
                   │
                   ▼
  ┌──────────────────────────────────────────────────────────┐
  │ LLM 寫:                                                   │
  │   - 預估本案賠償區間(低 / 中 / 高 估算)                    │
  │   - 預估過失比例 + 依據                                     │
  │   - 加重 / 對你有利的因素                                   │
  │   - 減輕 / 對你不利的因素                                   │
  │   - 蒐證清單(警察報告 / 醫療收據 / 工作損失證明 / ...)       │
  │   - 下一步實務建議(存證信函 → 調解 → 訴訟 3 步驟)           │
  │ **嚴禁保證會贏 / 不勸完全提告 / 不勸完全不提告**             │
  └──────────────────────────────────────────────────────────┘
```

3 個關鍵架構決策:

1. **兩階段 retrieval 是核心**:單靠 keyword 會把「機車**追撞**汽車」跟「機車**被**汽車追撞」混在一起;單靠 LLM 直接 rerank 又貴又慢(每次 query 都掃整個 corpus)。Stage 1 純函式快速過濾 + Stage 2 LLM 法律判斷 = 最佳組合。
2. **金額 100% 純函式**:賠償金額、過失比例、min/median/max、Top-3 statistics 全在 `retriever.py:compute_compensation_stats`;LLM 收結果後**只引用,不重算**。
3. **責任分清楚**:「**caselens 不是法律意見**」這句在每份報告底部固定出現;LLM 嚴禁說「會贏」「會輸」;建議路徑是 **存證信函 → 調解委員會(免費)→ 訴訟**,而非「**直接去找律師**」(避免引流到貴的法律服務)。

## 動作

### 純函式檢索(免 API key)

```bash
python3 caselens.py --db samples/cases_db.json --query samples/user_query.txt --no-ai --out output.md
```

`samples/user_query.txt` 是一則「機車正常綠燈被汽車後追撞 + 揮鞭症候群」情境描述。`samples/cases_db.json` 含 15 個 mock 車禍判決案例。

輸出含 Top 8 候選 + Top 3 詳細(含賠償組成)+ 純函式 Top 3 賠償統計。

### 兩階段 AI 模式(需 API key)

```bash
pip install -r requirements.txt
export ANTHROPIC_API_KEY=sk-ant-...
python3 caselens.py --db samples/cases_db.json --query samples/user_query.txt --out output.md
```

加上 Claude 做 Stage 2 法律相關性 re-rank + 寫:預估賠償區間 + 過失比例預估 + 加重 / 減輕因子 + 蒐證清單 + 下一步建議。

### 預先產出的 demo

`examples/sample_output.md` 是 AI 模式完整報告,展示:
- Stage 2 re-rank 把 TPHV-114-A-011(原告被追撞)排第 1(Stage 1 keyword 排第 1 但分數打平 KSHV-114-A-008 原告追撞別人)
- **明確排除** KSHV-114-A-008,因「角色方向相反」雖然 keyword 相似度高
- 預估賠償區間 NT$23-45 萬 / 中位 NT$32 萬
- 預估過失比例 0/100
- 8 條蒐證清單 + 3 步驟下一步建議(存證信函 → 調解 → 訴訟)

`examples/sample_output_no_ai.md` 是純函式模式,證明 Stage 1 已可用但 Stage 2 提供顯著價值。

## 真正產品要有但 prototype 沒做

- **接司法院裁判書 API**:即時抓取最新判決,而非 mock corpus
- **dense embedding 取代 Jaccard**:用 BAAI/bge-zh-v1.5 或 OpenAI text-embedding 算 dense vector;Jaccard 是 fallback
- **多類別擴展**:目前 corpus 只有車禍;真實版含鄰居糾紛 / 勞資爭議 / 房東房客 / 醫療糾紛 / 消費糾紛 / 信用卡 / 公寓大廈 等 10+ 類
- **判決趨勢分析**:同類案件「近 5 年判決金額變化曲線」(精神慰撫金實務逐年上漲)
- **法官實名分析(慎)**:同類案件「不同法官 / 法院賠償金額分布」(學術用,商業用需法律意見)
- **存證信函 / 起訴狀草稿生成**:整合 sudoc(r8)做下游 doc-gen
- **法扶基金會 / 調解委員會 預約整合**:直接幫使用者預約諮詢時段
- **訴訟費用試算機**:NT$X 求償金額 → 裁判費 + 律師費 + 時間 全試算

## 風險評估

| 風險 | 評估 | 緩解 |
|------|------|------|
| **被指控提供法律意見(無律師資格)** | 高 | UI / 報告底部多次強調「不是法律意見」+「請洽律師」+ 不下勝訴 / 敗訴斷論;與律師 / 法扶協會合作背書 |
| **判決資料時效 / 完整度** | 中 | 即時接司法院 API + 每日更新;明確標示「2024 年判決」 |
| **使用者過度依賴 AI 估算的賠償區間** | 中 | UI 強調區間 + 多次「實際視證據 / 律師而定」disclaimer |
| **律師反感(搶飯碗)** | 中 | 走 partner 路線(律所 Studio 方案 / 推薦律師費)而非競爭 |
| **保險公司反感** | 高 | 走 partner 路線(保險公司理賠部門白標)反而幫他們做客觀依據 |
| **個資保護(用戶上傳案件詳情)** | 高 | stateless API call;查詢不上雲;企業版 self-host LLM |
| **司法院 API 變更 / 收費** | 中 | 純公開判決書可合法使用;若 API 收費可考慮自建爬蟲(法律已認可) |
